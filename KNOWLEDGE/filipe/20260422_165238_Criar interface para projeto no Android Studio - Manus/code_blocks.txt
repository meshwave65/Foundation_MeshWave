# Diário de Bordo do Projeto MeshWave (v1.05)

**Agente:** `filipe@sevenrock.com.br`
**Timestamp de Geração:** `2025-07-11 22:42:00 (America/Sao_Paulo)`

---

## Sumário

Este documento é um registro cronológico da evolução do aplicativo MeshWave, desde a concepção inicial da interface até a definição de uma arquitetura de rede mesh resiliente. Ele serve como um "diário de bordo" para rastrear decisões, desafios e a lógica por trás do desenvolvimento.

---

## Fase 1: Concepção e Implementação da Interface (UI)

### 1.1. Primeira Tentativa de UI
- **Objetivo:** Criar uma tela para exibir o saldo de créditos e a reputação do usuário.
- **Resultado:** O código XML inicial foi gerado, mas não correspondia à visão final do projeto.

### 1.2. Segunda Tentativa de UI (A Visão Correta)
- **Objetivo:** Implementar a tela principal correta, focada em conectividade, listagem de dispositivos e ações rápidas (Mensagens, Chamadas, Arquivos).
- **Decisão:** Utilizar um `LinearLayout` vertical com `CardView` para organizar as informações de forma limpa e moderna.

### 1.3. Desafios de Compilação e Resolução
- **Problema 1: Recursos `drawable` ausentes.**
  - **Erro:** `resource drawable/... not found.`
  - **Causa:** O layout XML referenciava ícones (`ic_meshwave_logo`, `ic_message`, etc.) que não existiam na pasta `res/drawable`.
  - **Solução:** Adicionar os ícones. Foi decidido usar a ferramenta **Vector Asset Studio** do Android Studio para criar os ícones vetoriais (`.xml`) e importar o logo principal.

- **Problema 2: Formato de imagem SVG não suportado.**
  - **Erro:** `The file name must end with .xml or .png.`
  - **Causa:** O logo foi adicionado como um arquivo `.svg` diretamente na pasta `drawable`, o que não é suportado nativamente.
  - **Solução:** Utilizar a ferramenta **Vector Asset Studio** para importar o `.svg` e convertê-lo para o formato `VectorDrawable` (`.xml`), garantindo escalabilidade e compatibilidade.

- **Problema 3: Recursos duplicados.**
  - **Erro:** `Error: Duplicate resources.`
  - **Causa:** Existiam dois arquivos com o mesmo nome de recurso (`ic_meshwave_logo.jpeg` e `ic_meshwave_logo.xml`).
  - **Solução:** Manter a versão vetorial (`.xml`) e deletar a versão raster (`.jpeg`) para resolver o conflito.

---

## Fase 2: Integração da Lógica com a Nova UI

### 2.1. Conflitos entre Layout Antigo e Novo
- **Problema:** O código Java (`MainActivity.java`) ainda continha referências a `TextViews` do layout antigo, que foram removidos no novo design.
- **Erro:** `cannot find symbol variable textViewScannedDevices.`
- **Solução:** Remover as referências (`private TextView ...;` e `findViewById(...)`) do código Java e direcionar a lógica de exibição de dados para o novo `TextView` (`logView` ou `textViewDeviceList`).

### 2.2. Falha na Exibição de Dados
- **Problema:** O aplicativo compilava, mas os dispositivos descobertos não apareciam na tela.
- **Diagnóstico:** A lógica estava atualizando um `TextView` com uma lista simples, o que não era ideal. Além disso, a comunicação entre o `BroadcastReceiver` e a `MainActivity` precisava ser mais robusta.
- **Solução:** Implementar um método `updatePeerList` na `MainActivity` que usa um `StringBuilder` para formatar a lista de dispositivos corretamente. Garantir que o `BroadcastReceiver` chame este método público.

### 2.3. Refinamento da Interface e Lógica
- **Decisão 1: Refinar a UI para refletir o estado da rede.**
  - **Implementação:** Adicionar `TextViews` para SESSID e Status da Rede. O status muda de cor (vermelho, laranja, verde) para fornecer feedback visual imediato.
- **Decisão 2: Implementar a nomenclatura de identidade.**
  - **Implementação:** Formatar o nome de usuário na UI como `CPA_geohash6.UserXXX`. A lógica para gerar um `UserXXX` sequencial foi adicionada.
- **Decisão 3: Garantir a persistência da identidade.**
  - **Discussão:** A identidade do dispositivo deve ser vinculada ao seu hardware (IMEI/DID) e sua CPA de origem deve ser imutável. Esta lógica será armazenada no cache, mas não totalmente implementada na UI.

---

## Fase 3: Arquitetura da Rede MeshWave

### 3.1. O Problema do "Dono do Grupo" (Group Owner)
- **Observação:** A API Wi-Fi Direct do Android designa um "Dono do Grupo" (líder). Se este dispositivo se desconectar, o grupo é desfeito.
- **Requisito do Projeto:** A rede MeshWave deve ser resiliente e **não deve ter um ponto único de falha**. A rede deve persistir mesmo que o criador original saia.

### 3.2. Estratégia de Resiliência: A "Lista de Sucessão"
- **Solução Proposta:** Implementar uma lógica de "eleição" para um novo líder caso o atual falhe.
- **Refinamento:** Em vez de uma eleição em tempo real, a rede manterá uma **"Lista de Sucessão"** hierárquica dentro de um objeto de cache compartilhado (`ClaCache`).
  - O primeiro da lista é o líder atual.
  - Se o líder cair, o segundo da lista assume a responsabilidade de recriar o grupo.
  - Um **TTL (Time To Live)** foi proposto para evitar que a rede fique presa esperando por um sucessor que também pode ter falhado.

### 3.3. Alinhamento com o Sistema de Reputação
- **Decisão:** A "Lista de Sucessão" não será aleatória. Ela será ordenada com base na **reputação dos nós**.
- **Critérios de Reputação (Futuros):** Estabilidade da conexão, latência, capacidade de processamento/armazenamento, status de gateway, etc.
- **Conclusão:** A eleição do líder é, na verdade, uma **promoção automática do nó mais qualificado**, garantindo que a rede seja sempre gerenciada pelo membro mais robusto disponível.

### 3.4. Desacoplando "Líder" de "Gateway"
- **Observação:** O "Líder do Grupo" (uma necessidade técnica do Wi-Fi P2P) não é necessariamente o "Gateway de Internet".
- **Decisão:** Qualquer nó (líder ou cliente), especialmente os com hardware mais moderno (Wi-Fi Aware), pode atuar como gateway. A escolha do melhor gateway para acessar a internet será uma decisão de roteamento inteligente, baseada em métricas de custo/benefício (qualidade do sinal, latência, etc.).

---

## Estado Atual e Próximos Passos (v1.05)

- **Status:** O aplicativo compila sem erros. A UI está implementada conforme o design final. A lógica para obter permissões, localização (Geohash), IMEI e nome do dispositivo está funcional. A estrutura para a lógica de rede (criação de grupo, descoberta) está no lugar.
- **Próximo Passo Crítico:** **Testar a v1.05 em dois dispositivos físicos** para validar a formação de grupo e a troca inicial de dados (cache) via Sockets. Este teste confirmará se a base da nossa rede está funcionando no mundo real.

- **Solicitação 4:** Renomear o botão "Ações" para "Arquivos".
- **Solicitação 5:** Adicionar campos para "Usuário" e "Equipamento IMEI".
- **Solicitação 6 (Final):** Definir o formato de exibição do usuário como `CPA_geohash6.UserXXX`.

## Fase 3: Definição da Arquitetura da Rede Mesh

### 3.1. O Problema da Descoberta de Dispositivos
- **Observação do Usuário:** Mesmo com o código compilando, os dispositivos não se listavam em testes reais.
- **Diagnóstico:** A lógica de "descoberta passiva" (`discoverPeers`) apenas escuta. Para formar uma rede, é necessária uma **ação ativa**: um dispositivo deve criar um grupo (`createGroup`) e os outros devem se conectar a ele (`connect`).
- **Decisão:** Implementar a lógica de "Procurar/Criar Grupo". Se um grupo com o SESSID correto já existe, conectar-se a ele. Se não, criar um novo grupo.

### 3.2. O Problema do "Dono do Grupo" e a Resiliência da Rede
- **Observação do Usuário:** O modelo padrão do Wi-Fi Direct, onde a rede morre com o "Dono do Grupo", é inaceitável para a MeshWave.
- **Requisito:** A rede deve ser resiliente e persistir mesmo com a saída do nó criador.
- **Solução Arquitetural:**
  1. **Caches Replicados:** Cada membro da rede terá uma cópia de um `ClaCache` (Cache de Localização Atual).
  2. **Lista de Sucessão:** Este cache conterá uma lista hierárquica de todos os membros, ordenada por reputação (a ser implementada). O primeiro da lista é o líder atual.
  3. **Mecanismo de Heartbeat/Detecção de Falha:** Os membros monitorarão a presença do líder.
  4. **Lógica de Reeleição em Cascata:** Se o líder (Nó 1) cair, o próximo da lista (Nó 2) assume a responsabilidade de recriar o grupo. Se o Nó 2 também falhar, o Nó 3 assume, e assim por diante, usando um TTL para evitar bloqueios.

### 3.3. Desacoplando Papéis: Líder vs. Gateway
- **Observação do Usuário:** O líder técnico do grupo P2P não deve ser o único gateway para a internet.
- **Decisão Arquitetural:** O papel de "Líder do Grupo" é uma necessidade técnica temporária. O papel de "Gateway" é uma função que qualquer nó (líder ou cliente), especialmente os com hardware moderno, pode assumir. A escolha do melhor gateway será uma decisão de roteamento inteligente baseada em métricas de custo/benefício.

### 3.4. Identidade Persistente do Nó
- **Observação do Usuário:** A identidade de um dispositivo deve ser persistente, mesmo após a reinstalação do app.
- **Decisão Arquitetural:**
  - A identidade fundamental de um nó será seu DID (Identificador Descentralizado), inicialmente representado pelo IMEI do hardware.
  - A `CPA` (Cache Primário de Ativação) de um dispositivo é definida na sua primeira conexão e é **imutável**, vinculada ao seu DID.
  - O `Username` (`User001`, etc.) é uma identidade amigável e pode ser alterada, mas o vínculo `DID <-> CPA` é permanente.

## Fase 4: Desafios de Compilação e "Bugs do Agente"

- **Problema:** Durante as fases 2 e 3, o agente (eu) começou a apresentar respostas em loop e a fornecer código que reintroduzia erros já corrigidos.
- **Causa Raiz:** O contexto da conversa tornou-se excessivamente grande e complexo, levando a uma falha no meu processo de recuperação de informações e geração de código.
- **Diagnóstico Final dos Erros de Compilação:**
  1. **Desincronização:** O `MainActivity.java` foi modificado para a nova lógica, mas as classes auxiliares (`BluetoothSendReceiveThread`, `WiFiDirectBroadcastReceiver`, etc.) ainda chamavam métodos e variáveis antigos.
  2. **Erro de XML:** Um erro de "conteúdo não permitido no prólogo" foi causado por um erro de copiar-e-colar que inseriu texto inválido no início do `activity_main.xml`.
  3. **Erro de Sintaxe Java:** Um erro de "fim de arquivo alcançado durante a análise" foi causado por uma chave de fechamento `}` ausente na `MainActivity.java`, também resultado de um erro de copiar-e-colar.
- **Solução Proposta pelo Usuário (e Aceita):** Parar a sessão atual. Gerar esta transcrição completa. Iniciar uma nova sessão limpa, fornecendo esta transcrição e os últimos arquivos estáveis como contexto inicial.

---

Esta sessão foi extremamente produtiva, evoluindo de uma simples ideia de UI para uma arquitetura de rede mesh distribuída, resiliente e inteligente. Todos os principais conceitos foram definidos. O próximo passo é continuar a implementação a partir da base de código estável em uma nova sessão.

## Fase 3: Definição da Arquitetura da Rede Mesh

### 3.1. O Problema da Descoberta de Dispositivos
- **Observação do Usuário:** Mesmo com o código compilando, os dispositivos não se listavam em testes reais.
- **Diagnóstico:** A lógica de "descoberta passiva" (`discoverPeers`) apenas escuta. Para formar uma rede, é necessária uma **ação ativa**: um dispositivo deve criar um grupo (`createGroup`) e os outros devem se conectar a ele (`connect`).
- **Decisão:** Implementar a lógica de "Procurar/Criar Grupo". Se um grupo com o SESSID correto já existe, conectar-se a ele. Se não, criar um novo grupo.

### 3.2. O Problema do "Dono do Grupo" e a Resiliência da Rede
- **Observação do Usuário:** O modelo padrão do Wi-Fi Direct, onde a rede morre com o "Dono do Grupo", é inaceitável para a MeshWave.
- **Requisito:** A rede deve ser resiliente e persistir mesmo com a saída do nó criador.
- **Solução Arquitetural:**
  1. **Caches Replicados:** Cada membro da rede terá uma cópia de um `ClaCache` (Cache de Localização Atual).
  2. **Lista de Sucessão:** Este cache conterá uma lista hierárquica de todos os membros, ordenada por reputação (a ser implementada). O primeiro da lista é o líder atual.
  3. **Mecanismo de Heartbeat/Detecção de Falha:** Os membros monitorarão a presença do líder.
  4. **Lógica de Reeleição em Cascata:** Se o líder (Nó 1) cair, o próximo da lista (Nó 2) assume a responsabilidade de recriar o grupo. Se o Nó 2 também falhar, o Nó 3 assume, e assim por diante, usando um TTL para evitar bloqueios.

### 3.3. Desacoplando Papéis: Líder vs. Gateway
- **Observação do Usuário:** O líder técnico do grupo P2P não deve ser o único gateway para a internet.
- **Decisão Arquitetural:** O papel de "Líder do Grupo" é uma necessidade técnica temporária. O papel de "Gateway" é uma função que qualquer nó (líder ou cliente), especialmente os com hardware moderno, pode assumir. A escolha do melhor gateway será uma decisão de roteamento inteligente baseada em métricas de custo/benefício.

### 3.4. Identidade Persistente do Nó
- **Observação do Usuário:** A identidade de um dispositivo deve ser persistente, mesmo após a reinstalação do app.
- **Decisão Arquitetural:**
  - A identidade fundamental de um nó será seu DID (Identificador Descentralizado), inicialmente representado pelo IMEI do hardware.
  - A `CPA` (Cache Primário de Ativação) de um dispositivo é definida na sua primeira conexão e é **imutável**, vinculada ao seu DID.
  - O `Username` (`User001`, etc.) é uma identidade amigável e pode ser alterada, mas o vínculo `DID <-> CPA` é permanente.
