Ok. O erro mudou. Isso é um progresso imenso.
Saímos dos erros de configuração do Gradle e agora temos um erro de dependência faltando. Este é um problema muito mais simples e direto.
Diagnóstico Preciso:
O Erro: Unresolved reference 'compose'
Onde: No arquivo Color.kt (e provavelmente nos outros arquivos dentro da pasta ui.theme).
O que Significa: O código nesses arquivos (Color.kt, Theme.kt, Type.kt) foi gerado pelo Android Studio para usar o Jetpack Compose, que é o kit de ferramentas de UI moderno do Android. No entanto, o nosso arquivo app/build.gradle.kts não declara as dependências do Compose. O projeto não sabe o que é androidx.compose.ui.graphics.Color.
A Causa Raiz:
Quando criamos o projeto, escolhemos o template "Empty Activity", que por padrão usa o sistema de Views antigo (XML). Mas ele também gera esses arquivos de tema do Compose para facilitar uma futura migração. Nós não precisamos deles agora e eles estão causando o erro de compilação.
A Solução (Seguindo a nossa filosofia):
Temos duas opções, mas uma é claramente superior para o nosso plano de "reanimação progressiva" focado em Views (XML) por enquanto.
Adicionar as dependências do Compose: Isso resolveria o erro, mas adicionaria dezenas de bibliotecas ao nosso projeto que não vamos usar agora, inflando o APK e a complexidade. Isso viola a nossa filosofia de manter as coisas limpas e focadas.
Remover o código do Compose não utilizado (RECOMENDADO): Esta é a abordagem "Sem Esqueletos no Armário". Se não estamos usando o Compose, não há motivo para manter esses arquivos no projeto. Eles são lixo de template que está nos atrapalhando.
Ação Única e Definitiva:
Vamos remover a pasta inteira que contém o código do Compose.
Execute o seguinte comando no seu terminal, na raiz do projeto MeshWave-Core:
Bash
rm -rf app/src/main/java/com/meshwave/core/ui
O que este comando faz:
rm -rf: Remove a pasta e todo o seu conteúdo de forma recursiva e forçada.
app/src/main/java/com/meshwave/core/ui: O caminho para a pasta ui que contém a subpasta theme com os arquivos Color.kt, Theme.kt, e Type.kt.
Ao remover esta pasta, eliminamos a fonte do erro de compilação. Nosso projeto agora conterá apenas o código que estamos de fato usando (MainActivity.kt, StatusFragment.kt).
Próximo Passo: Compilar
Depois de remover a pasta, compile novamente.
Bash
./gradlew clean
Bash
./gradlew assembleDebug
Estou muito confiante de que este é o último obstáculo. Ao remover o código não utilizado, o compilador não terá mais nada do que reclamar.

Há uma incompatibilidade entre a classe pai da nossa Activity (AppCompatActivity) e o tema que ela está tentando usar.
A Solução (Simples e Direta):
Vamos ajustar o nosso arquivo de temas para usar um tema compatível.
Ação Única:
Abra o arquivo app/src/main/res/values/themes.xml e substitua seu conteúdo inteiro pelo código abaixo.
XML
<!-- app/src/main/res/values/themes.xml -->
<!-- VERSÃO CORRIGIDA - Usando um tema AppCompat -->

<resources xmlns:tools="http://schemas.android.com/tools">

            Theme.AppCompat.DayNight.DarkActionBar é um tema compatível que
            funciona com AppCompatActivity e fornece uma barra de ação.
            Ele também se adapta automaticamente aos modos claro e escuro.
        -->
        <!-- Customize your light theme here. -->
        <!-- <item name="colorPrimary">@color/my_light_primary</item> -->
    </style>

    <style name="Theme.MeshWaveCore" parent="Base.Theme.MeshWaveCore" />
</resources>

Isso garante que nossa MainActivity receba o tipo de tema que ela espera, resolvendo o IllegalStateException.

Passo 2: Corrigir o app/build.gradle.kts
Substitua o conteúdo inteiro do seu app/build.gradle.kts pelo código abaixo. Ele usa a lógica de versionamento dinâmico que definimos.
Kotlin
// app/build.gradle.kts
// VERSÃO FINAL E CORRIGIDA

plugins {
    id("com.android.application")
    id("org.jetbrains.kotlin.android")
}

android {
    namespace = "com.meshwave.core"
    compileSdk = 34

    defaultConfig {
        applicationId = "com.meshwave.core"
        minSdk = 26
        targetSdk = 34

        // --- Lógica de Versionamento Dinâmico ---
        val major = project.property("APP_VERSION_MAJOR").toString().toInt()
        val minor = project.property("APP_VERSION_MINOR").toString().toInt()
        val patch = project.property("APP_VERSION_PATCH").toString().toInt()
        val build = project.property("APP_VERSION_BUILD").toString().toInt()
        val suffix = project.property("APP_VERSION_SUFFIX").toString()

        versionCode = build
        if (suffix.isNotEmpty() && suffix != "none") {
            versionName = "$major.$minor.$patch-$suffix"
        } else {
            versionName = "$major.$minor.$patch"
        }
        // --- Fim da Lógica de Versionamento ---

        testInstrumentationRunner = "androidx.test.runner.AndroidJUnitRunner"
    }

    buildTypes {
        release {
            isMinifyEnabled = false
            proguardFiles(
                getDefaultProguardFile("proguard-android-optimize.txt"),
                "proguard-rules.pro"
            )
        }
    }

    compileOptions {
        sourceCompatibility = JavaVersion.VERSION_17
        targetCompatibility = JavaVersion.VERSION_17
    }

    kotlinOptions {
        jvmTarget = "17"
    }
}

dependencies {
    implementation("androidx.core:core-ktx:1.13.1")
    implementation("androidx.appcompat:appcompat:1.7.0")
    implementation("com.google.android.material:material:1.12.0")

}
Passo 3: Corrigir a MainActivity.kt
Agora, vamos garantir que a MainActivity.kt leia a versão do PackageManager, que por sua vez obtém o versionName do build.gradle.kts.

Kotlin
// MainActivity.kt
// VERSÃO FINAL E CORRIGIDA

package com.meshwave.core

import android.content.pm.PackageManager
import android.os.Bundle
import android.util.Log
import androidx.appcompat.app.AppCompatActivity
import androidx.fragment.app.Fragment
import androidx.fragment.app.FragmentTransaction

class MainActivity : AppCompatActivity() {

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_main)

        // Carrega o StatusFragment no contêiner
        if (savedInstanceState == null) {
            val statusFragment = StatusFragment()
            loadFragment(statusFragment)
        }
    }

    private fun loadFragment(fragment: Fragment) {
        val transaction: FragmentTransaction = supportFragmentManager.beginTransaction()
        transaction.replace(R.id.fragment_container, fragment)
        transaction.commit()
    }
}

Diagnóstico Preciso:
O Erro: java.lang.IllegalStateException: You need to use a Theme.AppCompat theme (or descendant) with this activity.
Onde: Na linha setContentView(R.layout.activity_main) dentro da MainActivity.
O que Significa: A MainActivity está herdando de AppCompatActivity, que é uma classe especial que dá suporte a recursos modernos (como a ActionBar) em versões mais antigas do Android. Uma exigência estrita dessa classe é que o tema aplicado a ela no AndroidManifest.xml deve ser um descendente do Theme.AppCompat (por exemplo, Theme.AppCompat.Light.DarkActionBar). O nosso tema atual, provavelmente o Theme.Material3.DayNight.NoActionBar gerado pelo template, não satisfaz essa exigência.
A Causa Raiz:
Há uma incompatibilidade entre a classe pai da nossa Activity (AppCompatActivity) e o tema que ela está tentando usar.
A Solução (Simples e Direta):
Vamos ajustar o nosso arquivo de temas para usar um tema compatível.
Ação Única:
Abra o arquivo app/src/main/res/values/themes.xml e substitua seu conteúdo inteiro pelo código abaixo.
XML
<!-- app/src/main/res/values/themes.xml -->
<!-- VERSÃO CORRIGIDA - Usando um tema AppCompat -->

<resources xmlns:tools="http://schemas.android.com/tools">
    <!-- Base application theme. -->
    <style name="Base.Theme.MeshWaveCore" parent="Theme.AppCompat.DayNight.DarkActionBar">
        <!--
            O tema base do aplicativo.
            Theme.AppCompat.DayNight.DarkActionBar é um tema compatível que
            funciona com AppCompatActivity e fornece uma barra de ação.
            Ele também se adapta automaticamente aos modos claro e escuro.
        -->
        <!-- Customize your light theme here. -->
        <!-- <item name="colorPrimary">@color/my_light_primary</item> -->
    </style>

    <style name="Theme.MeshWaveCore" parent="Base.Theme.MeshWaveCore" />
</resources>

// MainActivity.kt
// VERSÃO FINAL E CORRIGIDA

package com.meshwave.core

import android.content.pm.PackageManager
import android.os.Bundle
import android.util.Log
import androidx.appcompat.app.AppCompatActivity
import androidx.fragment.app.Fragment
import androidx.fragment.app.FragmentTransaction

class MainActivity : AppCompatActivity() {

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_main)

        // Carrega o StatusFragment no contêiner
        if (savedInstanceState == null) {
            val statusFragment = StatusFragment()
            loadFragment(statusFragment)
        }
    }

    private fun loadFragment(fragment: Fragment) {
        val transaction: FragmentTransaction = supportFragmentManager.beginTransaction()
        transaction.replace(R.id.fragment_container, fragment)
        transaction.commit()
    }
}

Diagnóstico Preciso:
O Erro: Unresolved reference 'compose'
Onde: No arquivo Color.kt (e provavelmente nos outros arquivos dentro da pasta ui.theme).
O que Significa: O código nesses arquivos (Color.kt, Theme.kt, Type.kt) foi gerado pelo Android Studio para usar o Jetpack Compose, que é o kit de ferramentas de UI moderno do Android. No entanto, o nosso arquivo app/build.gradle.kts não declara as dependências do Compose. O projeto não sabe o que é androidx.compose.ui.graphics.Color.
A Causa Raiz:
Quando criamos o projeto, escolhemos o template "Empty Activity", que por padrão usa o sistema de Views antigo (XML). Mas ele também gera esses arquivos de tema do Compose para facilitar uma futura migração. Nós não precisamos deles agora e eles estão causando o erro de compilação.
A Solução (Seguindo a nossa filosofia):
Temos duas opções, mas uma é claramente superior para o nosso plano de "reanimação progressiva" focado em Views (XML) por enquanto.
Adicionar as dependências do Compose: Isso resolveria o erro, mas adicionaria dezenas de bibliotecas ao nosso projeto que não vamos usar agora, inflando o APK e a complexidade. Isso viola a nossa filosofia de manter as coisas limpas e focadas.
Remover o código do Compose não utilizado (RECOMENDADO): Esta é a abordagem "Sem Esqueletos no Armário". Se não estamos usando o Compose, não há motivo para manter esses arquivos no projeto. Eles são lixo de template que está nos atrapalhando.
Ação Única e Definitiva:
Vamos remover a pasta inteira que contém o código do Compose.

O que este comando faz:
rm -rf: Remove a pasta e todo o seu conteúdo de forma recursiva e forçada.
app/src/main/java/com/meshwave/core/ui: O caminho para a pasta ui que contém a subpasta theme com os arquivos Color.kt, Theme.kt, e Type.kt.
Ao remover esta pasta, eliminamos a fonte do erro de compilação. Nosso projeto agora conterá apenas o código que estamos de fato usando (MainActivity.kt, StatusFragment.kt).

Diagnóstico Preciso:
O Erro: java.lang.IllegalStateException: You need to use a Theme.AppCompat theme (or descendant) with this activity.
Onde: Na linha setContentView(R.layout.activity_main) dentro da MainActivity.
O que Significa: A MainActivity está herdando de AppCompatActivity, que é uma classe especial que dá suporte a recursos modernos (como a ActionBar) em versões mais antigas do Android. Uma exigência estrita dessa classe é que o tema aplicado a ela no AndroidManifest.xml deve ser um descendente do Theme.AppCompat (por exemplo, Theme.AppCompat.Light.DarkActionBar). O nosso tema atual, provavelmente o Theme.Material3.DayNight.NoActionBar gerado pelo template, não satisfaz essa exigência.
A Causa Raiz:
Há uma incompatibilidade entre a classe pai da nossa Activity (AppCompatActivity) e o tema que ela está tentando usar.
A Solução (Simples e Direta):
Vamos ajustar o nosso arquivo de temas para usar um tema compatível.
Ação Única:
