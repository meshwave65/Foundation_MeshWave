1. Objetivo Maior (Missão v0.3.x)
O objetivo principal e imutável para o marco v0.3.x é:
Estabelecer uma conexão de comunicação básica e estável entre dois dispositivos Android, um deles sendo um hardware de baixo recurso (Moto E5 Play, Android 8.0).
Este objetivo se desdobra em três critérios de sucesso claros:
Inicialização Estável: O aplicativo deve iniciar, obter as permissões necessárias, gerar uma identidade baseada em geolocalização e exibir seu estado na UI, sem travar.
Conexão de Rede: Os dois dispositivos devem se descobrir e estabelecer uma conexão de rede funcional, priorizando o Bluetooth como o canal mais provável de sucesso no hardware-alvo.
Comunicação Mínima: Um dispositivo deve ser capaz de enviar uma mensagem de texto simples, e o outro deve recebê-la e exibi-la na tela de chat.
2. Arquitetura Atual (Consolidada)
Após um longo processo de depuração, chegamos a uma arquitetura robusta que se tornou nosso padrão:
Arquitetura "Bootstrap Híbrido":
A MainActivity (componente de UI) é responsável pela tarefa para a qual é mais eficiente: solicitar permissões e obter a localização inicial do dispositivo.
Uma vez que a localização é obtida, ela "entrega" essa informação para o MeshWaveService e o inicia.
O MeshWaveService (componente de segundo plano) assume como o cérebro persistente da operação. Ele é responsável por toda a lógica de identidade, cache e, mais importante, pela gestão da rede (descoberta e conexão).
Benefício: Esta arquitetura resolveu os problemas de instabilidade (a conexão não morre com a UI) e os problemas de acesso a APIs (a Activity obtém a localização de forma mais confiável que o Service).
Arquitetura de Rede "Carrossel Anti-ANR":
Dentro do MeshWaveService, a inicialização dos subsistemas de rede (Wi-Fi Direct, Bluetooth) não ocorre de forma paralela ou com Handlers na thread principal.
Ela segue uma sequência lógica (um "carrossel") executada inteiramente em uma thread de trabalho (backgroundExecutor).
Fluxo: 1º Processa Identidade -> 2º Tenta Wi-Fi -> 3º Tenta Bluetooth -> 4º Estabiliza.
Benefício: Isso resolveu o problema de ANR (Application Not Responding) que ocorria quando as operações de rede bloqueavam a thread principal. Cada etapa é encapsulada e uma falha em uma delas (ex: Wi-Fi) não impede a tentativa da próxima (Bluetooth).

4. Conceitos Estratégicos e de Roadmap (Definidos pelo Usuário)
Durante o desenvolvimento, definimos os seguintes pilares estratégicos que guiarão o futuro do projeto:
Política de Versionamento (Metodologia v3.1):
O segundo nível da versão (ex: 0.3.x -> 0.4.x) só é incrementado após um Objetivo Maior ser cumprido e validado em hardware real.
O terceiro nível (ex: 0.3.20 -> 0.3.21) só é incrementado para marcar uma "Tentativa Efetiva", ou seja, a geração de um APK para um novo ciclo de testes em hardware real com uma nova abordagem ou correção significativa.
Identidade vs. Nickname:
A identidade fundamental de um nó é seu DID (Device ID), que é imutável e composto por ANDROID_ID e outros elementos futuros.
O Username é um apelido mutável e efêmero, associado ao DID no cache. Ele não faz parte da identidade central.
Hierarquia de Transporte e a "Lógica do Macaco Velho":
Visão de Longo Prazo: O sistema deve ser multicanal e resiliente.
Hierarquia (do menor para o maior custo): Bluetooth -> Wi-Fi Direct -> Wi-Fi Convencional (TCP/Gateway) -> GSM -> Satélite.
"Lógica do Macaco Velho": O sistema deve sempre tentar manter a conexão mais barata e eficiente possível. Ele nunca "larga um galho (conexão) sem estar seguro em outro". Isso implica em uma lógica de handover "make-before-break".
Roadmap: Este é o objetivo principal para a Missão v0.4.x e além.
Isolamento de Tecnologia ("Não Limitar o Moderno pelo Antigo"):
O sistema deve ser capaz de detectar as capacidades do hardware em que está rodando.
Um dispositivo moderno deve poder usar funcionalidades avançadas (ex: Wi-Fi Aware), enquanto um dispositivo antigo pode se limitar ao Bluetooth, sem que um restrinja o outro.
Roadmap: A implementação de um Gerenciador de Capacidades é uma tarefa para a Missão v0.5.x ou posterior.

1. Objetivo Maior (Missão v0.3.x)
O objetivo principal e imutável para o marco v0.3.x é:
Estabelecer uma conexão de comunicação básica e estável entre dois dispositivos Android, um deles sendo um hardware de baixo recurso (Moto E5 Play, Android 8.0).
Este objetivo se desdobra em três critérios de sucesso claros:
Inicialização Estável: O aplicativo deve iniciar, obter as permissões necessárias, gerar uma identidade baseada em geolocalização e exibir seu estado na UI, sem travar.
Conexão de Rede: Os dois dispositivos devem se descobrir e estabelecer uma conexão de rede funcional, priorizando o Bluetooth como o canal mais provável de sucesso no hardware-alvo.
Comunicação Mínima: Um dispositivo deve ser capaz de enviar uma mensagem de texto simples, e o outro deve recebê-la e exibi-la na tela de chat.
2. Arquitetura Atual (Consolidada)
Após um longo processo de depuração, chegamos a uma arquitetura robusta que se tornou nosso padrão:
Arquitetura "Bootstrap Híbrido":
A MainActivity (componente de UI) é responsável pela tarefa para a qual é mais eficiente: solicitar permissões e obter a localização inicial do dispositivo.
Uma vez que a localização é obtida, ela "entrega" essa informação para o MeshWaveService e o inicia.
O MeshWaveService (componente de segundo plano) assume como o cérebro persistente da operação. Ele é responsável por toda a lógica de identidade, cache e, mais importante, pela gestão da rede (descoberta e conexão).
Benefício: Esta arquitetura resolveu os problemas de instabilidade (a conexão não morre com a UI) e os problemas de acesso a APIs (a Activity obtém a localização de forma mais confiável que o Service).
Arquitetura de Rede "Carrossel Anti-ANR":
Dentro do MeshWaveService, a inicialização dos subsistemas de rede (Wi-Fi Direct, Bluetooth) não ocorre de forma paralela ou com Handlers na thread principal.
Ela segue uma sequência lógica (um "carrossel") executada inteiramente em uma thread de trabalho (backgroundExecutor).
Fluxo: 1º Processa Identidade -> 2º Tenta Wi-Fi -> 3º Tenta Bluetooth -> 4º Estabiliza.
Benefício: Isso resolveu o problema de ANR (Application Not Responding) que ocorria quando as operações de rede bloqueavam a thread principal. Cada etapa é encapsulada e uma falha em uma delas (ex: Wi-Fi) não impede a tentativa da próxima (Bluetooth).

4. Conceitos Estratégicos e de Roadmap (Definidos pelo Usuário)
Durante o desenvolvimento, definimos os seguintes pilares estratégicos que guiarão o futuro do projeto:
Política de Versionamento (Metodologia v3.1):
O segundo nível da versão (ex: 0.3.x -> 0.4.x) só é incrementado após um Objetivo Maior ser cumprido e validado em hardware real.
O terceiro nível (ex: 0.3.20 -> 0.3.21) só é incrementado para marcar uma "Tentativa Efetiva", ou seja, a geração de um APK para um novo ciclo de testes em hardware real com uma nova abordagem ou correção significativa.
Identidade vs. Nickname:
A identidade fundamental de um nó é seu DID (Device ID), que é imutável e composto por ANDROID_ID e outros elementos futuros.
O Username é um apelido mutável e efêmero, associado ao DID no cache. Ele não faz parte da identidade central.
Hierarquia de Transporte e a "Lógica do Macaco Velho":
Visão de Longo Prazo: O sistema deve ser multicanal e resiliente.
Hierarquia (do menor para o maior custo): Bluetooth -> Wi-Fi Direct -> Wi-Fi Convencional (TCP/Gateway) -> GSM -> Satélite.
"Lógica do Macaco Velho": O sistema deve sempre tentar manter a conexão mais barata e eficiente possível. Ele nunca "larga um galho (conexão) sem estar seguro em outro". Isso implica em uma lógica de handover "make-before-break".
Roadmap: Este é o objetivo principal para a Missão v0.4.x e além.
Isolamento de Tecnologia ("Não Limitar o Moderno pelo Antigo"):
O sistema deve ser capaz de detectar as capacidades do hardware em que está rodando.
Um dispositivo moderno deve poder usar funcionalidades avançadas (ex: Wi-Fi Aware), enquanto um dispositivo antigo pode se limitar ao Bluetooth, sem que um restrinja o outro.
Roadmap: A implementação de um Gerenciador de Capacidades é uma tarefa para a Missão v0.5.x ou posterior.

Após um longo processo de depuração, chegamos a uma arquitetura robusta que se tornou nosso padrão:
Arquitetura "Bootstrap Híbrido":
A MainActivity (componente de UI) é responsável pela tarefa para a qual é mais eficiente: solicitar permissões e obter a localização inicial do dispositivo.

Arquitetura de Rede "Carrossel Anti-ANR":
Dentro do MeshWaveService, a inicialização dos subsistemas de rede (Wi-Fi Direct, Bluetooth) não ocorre de forma paralela ou com Handlers na thread principal.
Ela segue uma sequência lógica (um "carrossel") executada inteiramente em uma thread de trabalho (backgroundExecutor).
Fluxo: 1º Processa Identidade -> 2º Tenta Wi-Fi -> 3º Tenta Bluetooth -> 4º Estabiliza.
Benefício: Isso resolveu o problema de ANR (Application Not Responding) que ocorria quando as operações de rede bloqueavam a thread principal. Cada etapa é encapsulada e uma falha em uma delas (ex: Wi-Fi) não impede a tentativa da próxima (Bluetooth).

1. Objetivo Maior (Missão v0.3.x)
O objetivo principal e imutável para o marco v0.3.x é:
Estabelecer uma conexão de comunicação básica e estável entre dois dispositivos Android, um deles sendo um hardware de baixo recurso (Moto E5 Play, Android 8.0).
Este objetivo se desdobra em três critérios de sucesso claros:
Inicialização Estável: O aplicativo deve iniciar, obter as permissões necessárias, gerar uma identidade baseada em geolocalização e exibir seu estado na UI, sem travar.
Conexão de Rede: Os dois dispositivos devem se descobrir e estabelecer uma conexão de rede funcional, priorizando o Bluetooth como o canal mais provável de sucesso no hardware-alvo.
Comunicação Mínima: Um dispositivo deve ser capaz de enviar uma mensagem de texto simples, e o outro deve recebê-la e exibi-la na tela de chat.
2. Arquitetura Atual (Consolidada)
Após um longo processo de depuração, chegamos a uma arquitetura robusta que se tornou nosso padrão:
Arquitetura "Bootstrap Híbrido":
A MainActivity (componente de UI) é responsável pela tarefa para a qual é mais eficiente: solicitar permissões e obter a localização inicial do dispositivo.
Uma vez que a localização é obtida, ela "entrega" essa informação para o MeshWaveService e o inicia.
O MeshWaveService (componente de segundo plano) assume como o cérebro persistente da operação. Ele é responsável por toda a lógica de identidade, cache e, mais importante, pela gestão da rede (descoberta e conexão).
Benefício: Esta arquitetura resolveu os problemas de instabilidade (a conexão não morre com a UI) e os problemas de acesso a APIs (a Activity obtém a localização de forma mais confiável que o Service).
Arquitetura de Rede "Carrossel Anti-ANR":
Dentro do MeshWaveService, a inicialização dos subsistemas de rede (Wi-Fi Direct, Bluetooth) não ocorre de forma paralela ou com Handlers na thread principal.
Ela segue uma sequência lógica (um "carrossel") executada inteiramente em uma thread de trabalho (backgroundExecutor).
Fluxo: 1º Processa Identidade -> 2º Tenta Wi-Fi -> 3º Tenta Bluetooth -> 4º Estabiliza.
Benefício: Isso resolveu o problema de ANR (Application Not Responding) que ocorria quando as operações de rede bloqueavam a thread principal. Cada etapa é encapsulada e uma falha em uma delas (ex: Wi-Fi) não impede a tentativa da próxima (Bluetooth).

4. Conceitos Estratégicos e de Roadmap (Definidos pelo Usuário)
Durante o desenvolvimento, definimos os seguintes pilares estratégicos que guiarão o futuro do projeto:
Política de Versionamento (Metodologia v3.1):
O segundo nível da versão (ex: 0.3.x -> 0.4.x) só é incrementado após um Objetivo Maior ser cumprido e validado em hardware real.
O terceiro nível (ex: 0.3.20 -> 0.3.21) só é incrementado para marcar uma "Tentativa Efetiva", ou seja, a geração de um APK para um novo ciclo de testes em hardware real com uma nova abordagem ou correção significativa.
Identidade vs. Nickname:
A identidade fundamental de um nó é seu DID (Device ID), que é imutável e composto por ANDROID_ID e outros elementos futuros.
O Username é um apelido mutável e efêmero, associado ao DID no cache. Ele não faz parte da identidade central.
Hierarquia de Transporte e a "Lógica do Macaco Velho":
Visão de Longo Prazo: O sistema deve ser multicanal e resiliente.
Hierarquia (do menor para o maior custo): Bluetooth -> Wi-Fi Direct -> Wi-Fi Convencional (TCP/Gateway) -> GSM -> Satélite.
"Lógica do Macaco Velho": O sistema deve sempre tentar manter a conexão mais barata e eficiente possível. Ele nunca "larga um galho (conexão) sem estar seguro em outro". Isso implica em uma lógica de handover "make-before-break".
Roadmap: Este é o objetivo principal para a Missão v0.4.x e além.
Isolamento de Tecnologia ("Não Limitar o Moderno pelo Antigo"):
O sistema deve ser capaz de detectar as capacidades do hardware em que está rodando.
Um dispositivo moderno deve poder usar funcionalidades avançadas (ex: Wi-Fi Aware), enquanto um dispositivo antigo pode se limitar ao Bluetooth, sem que um restrinja o outro.
Roadmap: A implementação de um Gerenciador de Capacidades é uma tarefa para a Missão v0.5.x ou posterior.

1. Objetivo Maior (Missão v0.3.x)
O objetivo principal e imutável para o marco v0.3.x é:
Estabelecer uma conexão de comunicação básica e estável entre dois dispositivos Android, um deles sendo um hardware de baixo recurso (Moto E5 Play, Android 8.0).
Este objetivo se desdobra em três critérios de sucesso claros:
Inicialização Estável: O aplicativo deve iniciar, obter as permissões necessárias, gerar uma identidade baseada em geolocalização e exibir seu estado na UI, sem travar.
Conexão de Rede: Os dois dispositivos devem se descobrir e estabelecer uma conexão de rede funcional, priorizando o Bluetooth como o canal mais provável de sucesso no hardware-alvo.
Comunicação Mínima: Um dispositivo deve ser capaz de enviar uma mensagem de texto simples, e o outro deve recebê-la e exibi-la na tela de chat.
2. Arquitetura Atual (Consolidada)
Após um longo processo de depuração, chegamos a uma arquitetura robusta que se tornou nosso padrão:
Arquitetura "Bootstrap Híbrido":
A MainActivity (componente de UI) é responsável pela tarefa para a qual é mais eficiente: solicitar permissões e obter a localização inicial do dispositivo.
Uma vez que a localização é obtida, ela "entrega" essa informação para o MeshWaveService e o inicia.
O MeshWaveService (componente de segundo plano) assume como o cérebro persistente da operação. Ele é responsável por toda a lógica de identidade, cache e, mais importante, pela gestão da rede (descoberta e conexão).
Benefício: Esta arquitetura resolveu os problemas de instabilidade (a conexão não morre com a UI) e os problemas de acesso a APIs (a Activity obtém a localização de forma mais confiável que o Service).
Arquitetura de Rede "Carrossel Anti-ANR":
Dentro do MeshWaveService, a inicialização dos subsistemas de rede (Wi-Fi Direct, Bluetooth) não ocorre de forma paralela ou com Handlers na thread principal.
Ela segue uma sequência lógica (um "carrossel") executada inteiramente em uma thread de trabalho (backgroundExecutor).
Fluxo: 1º Processa Identidade -> 2º Tenta Wi-Fi -> 3º Tenta Bluetooth -> 4º Estabiliza.
Benefício: Isso resolveu o problema de ANR (Application Not Responding) que ocorria quando as operações de rede bloqueavam a thread principal. Cada etapa é encapsulada e uma falha em uma delas (ex: Wi-Fi) não impede a tentativa da próxima (Bluetooth).

4. Conceitos Estratégicos e de Roadmap (Definidos pelo Usuário)
Durante o desenvolvimento, definimos os seguintes pilares estratégicos que guiarão o futuro do projeto:
Política de Versionamento (Metodologia v3.1):
O segundo nível da versão (ex: 0.3.x -> 0.4.x) só é incrementado após um Objetivo Maior ser cumprido e validado em hardware real.
O terceiro nível (ex: 0.3.20 -> 0.3.21) só é incrementado para marcar uma "Tentativa Efetiva", ou seja, a geração de um APK para um novo ciclo de testes em hardware real com uma nova abordagem ou correção significativa.
Identidade vs. Nickname:
A identidade fundamental de um nó é seu DID (Device ID), que é imutável e composto por ANDROID_ID e outros elementos futuros.
O Username é um apelido mutável e efêmero, associado ao DID no cache. Ele não faz parte da identidade central.
Hierarquia de Transporte e a "Lógica do Macaco Velho":
Visão de Longo Prazo: O sistema deve ser multicanal e resiliente.
Hierarquia (do menor para o maior custo): Bluetooth -> Wi-Fi Direct -> Wi-Fi Convencional (TCP/Gateway) -> GSM -> Satélite.
"Lógica do Macaco Velho": O sistema deve sempre tentar manter a conexão mais barata e eficiente possível. Ele nunca "larga um galho (conexão) sem estar seguro em outro". Isso implica em uma lógica de handover "make-before-break".
Roadmap: Este é o objetivo principal para a Missão v0.4.x e além.
Isolamento de Tecnologia ("Não Limitar o Moderno pelo Antigo"):
O sistema deve ser capaz de detectar as capacidades do hardware em que está rodando.
Um dispositivo moderno deve poder usar funcionalidades avançadas (ex: Wi-Fi Aware), enquanto um dispositivo antigo pode se limitar ao Bluetooth, sem que um restrinja o outro.
Roadmap: A implementação de um Gerenciador de Capacidades é uma tarefa para a Missão v0.5.x ou posterior.

Documento de Migração de Contexto e Consciência Situacional – Projeto MeshWave
Data de Consolidação: 14 de julho de 2025
Versão-Alvo Atual: v0.3.21-alpha
Último Estado do Código: Pronto para a "Tentativa Efetiva v0.3.21"
1. Objetivo Maior (Missão v0.3.x)
O objetivo principal e imutável para o marco v0.3.x é:
Estabelecer uma conexão de comunicação básica e estável entre dois dispositivos Android, um deles sendo um hardware de baixo recurso (Moto E5 Play, Android 8.0).
Este objetivo se desdobra em três critérios de sucesso claros:
Inicialização Estável: O aplicativo deve iniciar, obter as permissões necessárias, gerar uma identidade baseada em geolocalização e exibir seu estado na UI, sem travar.
Conexão de Rede: Os dois dispositivos devem se descobrir e estabelecer uma conexão de rede funcional, priorizando o Bluetooth como o canal mais provável de sucesso no hardware-alvo.
Comunicação Mínima: Um dispositivo deve ser capaz de enviar uma mensagem de texto simples, e o outro deve recebê-la e exibi-la na tela de chat.
2. Arquitetura Atual (Consolidada)
Após um longo processo de depuração, chegamos a uma arquitetura robusta que se tornou nosso padrão:
Arquitetura "Bootstrap Híbrido":
A MainActivity (componente de UI) é responsável pela tarefa para a qual é mais eficiente: solicitar permissões e obter a localização inicial do dispositivo.
Uma vez que a localização é obtida, ela "entrega" essa informação para o MeshWaveService e o inicia.
O MeshWaveService (componente de segundo plano) assume como o cérebro persistente da operação. Ele é responsável por toda a lógica de identidade, cache e, mais importante, pela gestão da rede (descoberta e conexão).
Benefício: Esta arquitetura resolveu os problemas de instabilidade (a conexão não morre com a UI) e os problemas de acesso a APIs (a Activity obtém a localização de forma mais confiável que o Service).
Arquitetura de Rede "Carrossel Anti-ANR":
Dentro do MeshWaveService, a inicialização dos subsistemas de rede (Wi-Fi Direct, Bluetooth) não ocorre de forma paralela ou com Handlers na thread principal.
Ela segue uma sequência lógica (um "carrossel") executada inteiramente em uma thread de trabalho (backgroundExecutor).
Fluxo: 1º Processa Identidade -> 2º Tenta Wi-Fi -> 3º Tenta Bluetooth -> 4º Estabiliza.
Benefício: Isso resolveu o problema de ANR (Application Not Responding) que ocorria quando as operações de rede bloqueavam a thread principal. Cada etapa é encapsulada e uma falha em uma delas (ex: Wi-Fi) não impede a tentativa da próxima (Bluetooth).
3. Estado Atual do Código e Próximo Passo Imediato
Última Versão Compilada: v0.3.21-alpha.
Estado do Build: BUILD SUCCESSFUL.
Funcionalidade Implementada:
O fluxo de permissões na MainActivity está estável.
A lógica de "Bootstrap Híbrido" está implementada.
O MeshWaveService implementa a arquitetura "Carrossel Anti-ANR".
A lógica de identidade, geohash e cache local está funcional.
A lógica de descoberta e conexão para Wi-Fi Direct e Bluetooth foi reativada e encapsulada dentro do "Carrossel".
Próximo Passo Imediato (Ação Pendente):
Gerar o APK da versão v0.3.21-alpha.
Instalar e testar nos dispositivos de hardware real (Moto E5 Play) para validar se a arquitetura "Carrossel Anti-ANR" finalmente cumpre o Objetivo Maior da v0.3.x.
4. Conceitos Estratégicos e de Roadmap (Definidos pelo Usuário)
Durante o desenvolvimento, definimos os seguintes pilares estratégicos que guiarão o futuro do projeto:
Política de Versionamento (Metodologia v3.1):
O segundo nível da versão (ex: 0.3.x -> 0.4.x) só é incrementado após um Objetivo Maior ser cumprido e validado em hardware real.
O terceiro nível (ex: 0.3.20 -> 0.3.21) só é incrementado para marcar uma "Tentativa Efetiva", ou seja, a geração de um APK para um novo ciclo de testes em hardware real com uma nova abordagem ou correção significativa.
Identidade vs. Nickname:
A identidade fundamental de um nó é seu DID (Device ID), que é imutável e composto por ANDROID_ID e outros elementos futuros.
O Username é um apelido mutável e efêmero, associado ao DID no cache. Ele não faz parte da identidade central.
Hierarquia de Transporte e a "Lógica do Macaco Velho":
Visão de Longo Prazo: O sistema deve ser multicanal e resiliente.
Hierarquia (do menor para o maior custo): Bluetooth -> Wi-Fi Direct -> Wi-Fi Convencional (TCP/Gateway) -> GSM -> Satélite.
"Lógica do Macaco Velho": O sistema deve sempre tentar manter a conexão mais barata e eficiente possível. Ele nunca "larga um galho (conexão) sem estar seguro em outro". Isso implica em uma lógica de handover "make-before-break".
Roadmap: Este é o objetivo principal para a Missão v0.4.x e além.
Isolamento de Tecnologia ("Não Limitar o Moderno pelo Antigo"):
O sistema deve ser capaz de detectar as capacidades do hardware em que está rodando.
Um dispositivo moderno deve poder usar funcionalidades avançadas (ex: Wi-Fi Aware), enquanto um dispositivo antigo pode se limitar ao Bluetooth, sem que um restrinja o outro.
Roadmap: A implementação de um Gerenciador de Capacidades é uma tarefa para a Missão v0.5.x ou posterior.
Este documento representa a "memória" do nosso projeto. Estou pronto para iniciar uma nova tarefa com esta consciência situacional e dar continuidade a partir do Próximo Passo Imediato definido.

Após um longo processo de depuração, chegamos a uma arquitetura robusta que se tornou nosso padrão:

Arquitetura "Bootstrap Híbrido":
A MainActivity (componente de UI) é responsável pela tarefa para a qual é mais eficiente: solicitar permissões e obter a localização inicial do dispositivo.
Uma vez que a localização é obtida, ela "entrega" essa informação para o MeshWaveService e o inicia.
O MeshWaveService (componente de segundo plano) assume como o cérebro persistente da operação. Ele é responsável por toda a lógica de identidade, cache e, mais importante, pela gestão da rede (descoberta e conexão).
Benefício: Esta arquitetura resolveu os problemas de instabilidade (a conexão não morre com a UI) e os problemas de acesso a APIs (a Activity obtém a localização de forma mais confiável que o Service).
Arquitetura de Rede "Carrossel Anti-ANR":
Dentro do MeshWaveService, a inicialização dos subsistemas de rede (Wi-Fi Direct, Bluetooth) não ocorre de forma paralela ou com Handlers na thread principal.
Ela segue uma sequência lógica (um "carrossel") executada inteiramente em uma thread de trabalho (backgroundExecutor).
Fluxo: 1º Processa Identidade -> 2º Tenta Wi-Fi -> 3º Tenta Bluetooth -> 4º Estabiliza.
Benefício: Isso resolveu o problema de ANR (Application Not Responding) que ocorria quando as operações de rede bloqueavam a thread principal. Cada etapa é encapsulada e uma falha em uma delas (ex: Wi-Fi) não impede a tentativa da próxima (Bluetooth).
