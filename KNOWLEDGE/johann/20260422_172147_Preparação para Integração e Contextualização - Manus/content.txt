Processo de pensamento
Você está absolutamente certo em procurar o config.yml. Essa é a maneira correta e profissional de gerenciar múltiplos serviços com o Cloudflare Tunnels. O fato de você não o encontrar significa que o túnel provavelmente está sendo executado em modo "ad-hoc" (linha de comando direta), o que explica por que ele só expõe um serviço por vez.
Vamos resolver isso de uma vez por todas, criando a configuração correta. Este processo lhe dará controle total para expor quantos serviços você quiser.
Plano de Ação: Configurando o Cloudflare Tunnels para Múltiplos Serviços
Vamos seguir um processo de 3 passos:
Localizar o Túnel: Encontrar o ID do túnel que já está em uso.
Criar o Arquivo de Configuração: Criar o config.yml e definir as regras para expor todos os seus frontends.
Atualizar o sentinel.py: Modificar o comando para que ele use o novo arquivo de configuração.
Passo 1: Localizar o ID do seu Túnel Existente
Primeiro, precisamos saber o nome ou o ID do túnel que o sentinel.py está tentando usar.
Abra um terminal e execute o seguinte comando:
Bash
cloudflared tunnel list
A saída será algo como:
Plain Text
ID                                   NAME        CREATED              CONNECTIONS
=====================================================================================
a1b2c3d4-e5f6-7890-1234-567890abcdef  sofia-dev   2025-09-20T18:00:00Z 2xDFW, 2xLAX
Anote o NOME do túnel (neste exemplo, sofia-dev). É este nome que usaremos.
Passo 2: Criar o Arquivo config.yml
Agora, vamos criar o arquivo de configuração que define todas as suas "regras de tráfego".
Navegue até o Diretório de Configuração: O local padrão para este arquivo é ~/.cloudflared/.
Bash
cd ~/.cloudflared/
(Se o diretório não existir, crie-o com mkdir ~/.cloudflared e depois entre nele).
Crie e Edite o Arquivo:
Bash
nano config.yml
Cole o Seguinte Conteúdo no Arquivo:
(Substitua sofia-dev pelo nome do seu túnel e seudominio.com pelo seu domínio real).
YAML
# Arquivo de configuração para o túnel 'sofia-dev'
# Define as regras de entrada para múltiplos serviços locais.

tunnel: sofia-dev  # <-- SUBSTITUA pelo NOME do seu túnel
credentials-file: /home/mesh/.cloudflared/a1b2c3d4-e5f6-7890-1234-567890abcdef.json # <-- O cloudflared encontra isso sozinho, mas é bom saber que existe.

# Regras de entrada (ingress rules)
ingress:
  # Regra 1: Frontend Principal (SOFIA)
  - hostname: sofia.seudominio.com  # <-- SUBSTITUA pelo seu subdomínio desejado
    service: http://localhost:5173

  # Regra 2: Frontend do Agente Condensador
  - hostname: condenser.seudominio.com # <-- SUBSTITUA pelo seu subdomínio desejado
    service: http://localhost:5174

  # Regra 3: Frontend do Aquômetro
  - hostname: aquometro.seudominio.com # <-- SUBSTITUA pelo seu subdomínio desejado
    service: http://localhost:5175

  # Regra 4: Frontend do Condenser PRO (o novo )
  - hostname: condenser-pro.seudominio.com # <-- SUBSTITUA pelo seu subdomínio desejado
    service: http://localhost:5176

  # Regra Final (Catch-all ): Deve ser a última regra.
  # Qualquer tráfego que não corresponda às regras acima receberá um erro 404.
  - service: http_status:404
Salve e Feche o Arquivo: Pressione Ctrl + X, depois Y e Enter.
Configure o DNS no Cloudflare: Para cada hostname que você definiu (ex: sofia.seudominio.com ), você precisa criar um registro CNAME no painel de DNS do seu Cloudflare, apontando para o ID do seu túnel.
Tipo: CNAME
Nome: sofia (ou condenser, aquometro, etc.)
Conteúdo: a1b2c3d4-e5f6-7890-1234-567890abcdef.cfargotunnel.com (substitua pelo seu ID de túnel seguido de .cfargotunnel.com).
Passo 3: Atualizar o sentinel.py
Agora, o passo final. Vamos dizer ao sentinel.py para parar de usar o comando antigo e passar a usar o nosso novo arquivo de configuração.
Abra o sentinel.py para edição.
Encontre a linha que define o comando do Cloudflare. Ela deve ser algo como:
Python
"tunnel": [str(CLOUDFLARED_PATH), "tunnel", "--no-autoupdate", "run", "--token", str(CLOUDFLARE_TOKEN)],
Substitua essa linha pela nova versão, que é muito mais simples:
Python
# VERSÃO ATUALIZADA - Usa o arquivo de configuração padrão
"tunnel": [str(CLOUDFLARED_PATH), "tunnel", "--no-autoupdate", "run"],
Por que funciona? Quando você executa cloudflared tunnel run sem especificar um nome ou token, ele automaticamente procura e usa o arquivo config.yml no diretório ~/.cloudflared/.
Finalizando
Salve o sentinel.py modificado.
Reinicie o serviço SOFIA para que ele use o novo comando:
Bash
sudo systemctl restart sofia.service
Resultado:
Após o reinício, o sentinel.py iniciará um único processo cloudflared. Esse processo lerá seu config.yml e começará a rotear o tráfego dos seus subdomínios públicos para as portas locais corretas (5173, 5174, 5175, 5176).
Você terá todos os seus frontends expostos de forma segura e organizada.

Plano de Ação: Configurando o Cloudflare Tunnels para Múltiplos Serviços
Vamos seguir um processo de 3 passos:
Localizar o Túnel: Encontrar o ID do túnel que já está em uso.
Criar o Arquivo de Configuração: Criar o config.yml e definir as regras para expor todos os seus frontends.
Atualizar o sentinel.py: Modificar o comando para que ele use o novo arquivo de configuração.
Passo 1: Localizar o ID do seu Túnel Existente
Primeiro, precisamos saber o nome ou o ID do túnel que o sentinel.py está tentando usar.
Abra um terminal e execute o seguinte comando:
Bash
cloudflared tunnel list
A saída será algo como:
Plain Text
ID                                   NAME        CREATED              CONNECTIONS
=====================================================================================
a1b2c3d4-e5f6-7890-1234-567890abcdef  sofia-dev   2025-09-20T18:00:00Z 2xDFW, 2xLAX
Anote o NOME do túnel (neste exemplo, sofia-dev). É este nome que usaremos.
Passo 2: Criar o Arquivo config.yml
Agora, vamos criar o arquivo de configuração que define todas as suas "regras de tráfego".
Navegue até o Diretório de Configuração: O local padrão para este arquivo é ~/.cloudflared/.
Bash
cd ~/.cloudflared/
(Se o diretório não existir, crie-o com mkdir ~/.cloudflared e depois entre nele).
Crie e Edite o Arquivo:
Bash
nano config.yml
Cole o Seguinte Conteúdo no Arquivo:
(Substitua sofia-dev pelo nome do seu túnel e seudominio.com pelo seu domínio real).
YAML
# Arquivo de configuração para o túnel 'sofia-dev'
# Define as regras de entrada para múltiplos serviços locais.

tunnel: sofia-dev  # <-- SUBSTITUA pelo NOME do seu túnel
credentials-file: /home/mesh/.cloudflared/a1b2c3d4-e5f6-7890-1234-567890abcdef.json # <-- O cloudflared encontra isso sozinho, mas é bom saber que existe.

# Regras de entrada (ingress rules)
ingress:
  # Regra 1: Frontend Principal (SOFIA)
  - hostname: sofia.seudominio.com  # <-- SUBSTITUA pelo seu subdomínio desejado
    service: http://localhost:5173

  # Regra 2: Frontend do Agente Condensador
  - hostname: condenser.seudominio.com # <-- SUBSTITUA pelo seu subdomínio desejado
    service: http://localhost:5174

  # Regra 3: Frontend do Aquômetro
  - hostname: aquometro.seudominio.com # <-- SUBSTITUA pelo seu subdomínio desejado
    service: http://localhost:5175

  # Regra 4: Frontend do Condenser PRO (o novo )
  - hostname: condenser-pro.seudominio.com # <-- SUBSTITUA pelo seu subdomínio desejado
    service: http://localhost:5176

  # Regra Final (Catch-all ): Deve ser a última regra.
  # Qualquer tráfego que não corresponda às regras acima receberá um erro 404.
  - service: http_status:404
Salve e Feche o Arquivo: Pressione Ctrl + X, depois Y e Enter.
Configure o DNS no Cloudflare: Para cada hostname que você definiu (ex: sofia.seudominio.com ), você precisa criar um registro CNAME no painel de DNS do seu Cloudflare, apontando para o ID do seu túnel.
Tipo: CNAME
Nome: sofia (ou condenser, aquometro, etc.)
Conteúdo: a1b2c3d4-e5f6-7890-1234-567890abcdef.cfargotunnel.com (substitua pelo seu ID de túnel seguido de .cfargotunnel.com).
Passo 3: Atualizar o sentinel.py
Agora, o passo final. Vamos dizer ao sentinel.py para parar de usar o comando antigo e passar a usar o nosso novo arquivo de configuração.
Abra o sentinel.py para edição.
Encontre a linha que define o comando do Cloudflare. Ela deve ser algo como:
Python
"tunnel": [str(CLOUDFLARED_PATH), "tunnel", "--no-autoupdate", "run", "--token", str(CLOUDFLARE_TOKEN)],
Substitua essa linha pela nova versão, que é muito mais simples:
Python
# VERSÃO ATUALIZADA - Usa o arquivo de configuração padrão
"tunnel": [str(CLOUDFLARED_PATH), "tunnel", "--no-autoupdate", "run"],
Por que funciona? Quando você executa cloudflared tunnel run sem especificar um nome ou token, ele automaticamente procura e usa o arquivo config.yml no diretório ~/.cloudflared/.
Finalizando
Salve o sentinel.py modificado.
Reinicie o serviço SOFIA para que ele use o novo comando:
Bash
sudo systemctl restart sofia.service
Resultado:
Após o reinício, o sentinel.py iniciará um único processo cloudflared. Esse processo lerá seu config.yml e começará a rotear o tráfego dos seus subdomínios públicos para as portas locais corretas (5173, 5174, 5175, 5176).
Você terá todos os seus frontends expostos de forma segura e organizada.

Plano de Ação: Configurando o Cloudflare Tunnels para Múltiplos Serviços
Vamos seguir um processo de 3 passos:
Localizar o Túnel: Encontrar o ID do túnel que já está em uso.
Criar o Arquivo de Configuração: Criar o config.yml e definir as regras para expor todos os seus frontends.
Atualizar o sentinel.py: Modificar o comando para que ele use o novo arquivo de configuração.
Passo 1: Localizar o ID do seu Túnel Existente

Bash

Plain Text

Anote o NOME do túnel (neste exemplo, sofia-dev). É este nome que usaremos.
Passo 2: Criar o Arquivo config.yml
Agora, vamos criar o arquivo de configuração que define todas as suas "regras de tráfego".
Navegue até o Diretório de Configuração: O local padrão para este arquivo é ~/.cloudflared/.
Bash
cd ~/.cloudflared/
(Se o diretório não existir, crie-o com mkdir ~/.cloudflared e depois entre nele).
Crie e Edite o Arquivo:
Bash
nano config.yml
Cole o Seguinte Conteúdo no Arquivo:
(Substitua sofia-dev pelo nome do seu túnel e seudominio.com pelo seu domínio real).
YAML
# Arquivo de configuração para o túnel 'sofia-dev'
# Define as regras de entrada para múltiplos serviços locais.

tunnel: sofia-dev  # <-- SUBSTITUA pelo NOME do seu túnel
credentials-file: /home/mesh/.cloudflared/a1b2c3d4-e5f6-7890-1234-567890abcdef.json # <-- O cloudflared encontra isso sozinho, mas é bom saber que existe.

# Regras de entrada (ingress rules)
ingress:
  # Regra 1: Frontend Principal (SOFIA)
  - hostname: sofia.seudominio.com  # <-- SUBSTITUA pelo seu subdomínio desejado
    service: http://localhost:5173

  # Regra 2: Frontend do Agente Condensador
  - hostname: condenser.seudominio.com # <-- SUBSTITUA pelo seu subdomínio desejado
    service: http://localhost:5174

  # Regra 3: Frontend do Aquômetro
  - hostname: aquometro.seudominio.com # <-- SUBSTITUA pelo seu subdomínio desejado
    service: http://localhost:5175

  # Regra 4: Frontend do Condenser PRO (o novo )
  - hostname: condenser-pro.seudominio.com # <-- SUBSTITUA pelo seu subdomínio desejado
    service: http://localhost:5176

  # Regra Final (Catch-all ): Deve ser a última regra.
  # Qualquer tráfego que não corresponda às regras acima receberá um erro 404.
  - service: http_status:404
Salve e Feche o Arquivo: Pressione Ctrl + X, depois Y e Enter.
Configure o DNS no Cloudflare: Para cada hostname que você definiu (ex: sofia.seudominio.com ), você precisa criar um registro CNAME no painel de DNS do seu Cloudflare, apontando para o ID do seu túnel.
Tipo: CNAME
Nome: sofia (ou condenser, aquometro, etc.)
Conteúdo: a1b2c3d4-e5f6-7890-1234-567890abcdef.cfargotunnel.com (substitua pelo seu ID de túnel seguido de .cfargotunnel.com).

Bash

Plain Text

Anote o NOME do túnel (neste exemplo, sofia-dev). É este nome que usaremos.
Passo 2: Criar o Arquivo config.yml
Agora, vamos criar o arquivo de configuração que define todas as suas "regras de tráfego".
Navegue até o Diretório de Configuração: O local padrão para este arquivo é ~/.cloudflared/.
Bash
cd ~/.cloudflared/
(Se o diretório não existir, crie-o com mkdir ~/.cloudflared e depois entre nele).
Crie e Edite o Arquivo:
Bash
nano config.yml
Cole o Seguinte Conteúdo no Arquivo:
(Substitua sofia-dev pelo nome do seu túnel e seudominio.com pelo seu domínio real).
YAML
# Arquivo de configuração para o túnel 'sofia-dev'
# Define as regras de entrada para múltiplos serviços locais.

tunnel: sofia-dev  # <-- SUBSTITUA pelo NOME do seu túnel
credentials-file: /home/mesh/.cloudflared/a1b2c3d4-e5f6-7890-1234-567890abcdef.json # <-- O cloudflared encontra isso sozinho, mas é bom saber que existe.

# Regras de entrada (ingress rules)
ingress:
  # Regra 1: Frontend Principal (SOFIA)
  - hostname: sofia.seudominio.com  # <-- SUBSTITUA pelo seu subdomínio desejado
    service: http://localhost:5173

  # Regra 2: Frontend do Agente Condensador
  - hostname: condenser.seudominio.com # <-- SUBSTITUA pelo seu subdomínio desejado
    service: http://localhost:5174

  # Regra 3: Frontend do Aquômetro
  - hostname: aquometro.seudominio.com # <-- SUBSTITUA pelo seu subdomínio desejado
    service: http://localhost:5175

  # Regra 4: Frontend do Condenser PRO (o novo )
  - hostname: condenser-pro.seudominio.com # <-- SUBSTITUA pelo seu subdomínio desejado
    service: http://localhost:5176

  # Regra Final (Catch-all ): Deve ser a última regra.
  # Qualquer tráfego que não corresponda às regras acima receberá um erro 404.
  - service: http_status:404
Salve e Feche o Arquivo: Pressione Ctrl + X, depois Y e Enter.
Configure o DNS no Cloudflare: Para cada hostname que você definiu (ex: sofia.seudominio.com ), você precisa criar um registro CNAME no painel de DNS do seu Cloudflare, apontando para o ID do seu túnel.
Tipo: CNAME
Nome: sofia (ou condenser, aquometro, etc.)
Conteúdo: a1b2c3d4-e5f6-7890-1234-567890abcdef.cfargotunnel.com (substitua pelo seu ID de túnel seguido de .cfargotunnel.com).
Passo 3: Atualizar o sentinel.py
Agora, o passo final. Vamos dizer ao sentinel.py para parar de usar o comando antigo e passar a usar o nosso novo arquivo de configuração.

Python
"tunnel": [str(CLOUDFLARED_PATH), "tunnel", "--no-autoupdate", "run", "--token", str(CLOUDFLARE_TOKEN)],

Python
# VERSÃO ATUALIZADA - Usa o arquivo de configuração padrão

Por que funciona? Quando você executa cloudflared tunnel run sem especificar um nome ou token, ele automaticamente procura e usa o arquivo config.yml no diretório ~/.cloudflared/.

Bash
sudo systemctl restart sofia.service

Anote o NOME do túnel (neste exemplo, sofia-dev). É este nome que usaremos.

Agora, vamos criar o arquivo de configuração que define todas as suas "regras de tráfego".
Navegue até o Diretório de Configuração: O local padrão para este arquivo é ~/.cloudflared/.
Bash
cd ~/.cloudflared/
(Se o diretório não existir, crie-o com mkdir ~/.cloudflared e depois entre nele).
Crie e Edite o Arquivo:
Bash
nano config.yml
Cole o Seguinte Conteúdo no Arquivo:
(Substitua sofia-dev pelo nome do seu túnel e seudominio.com pelo seu domínio real).
YAML
# Arquivo de configuração para o túnel 'sofia-dev'
# Define as regras de entrada para múltiplos serviços locais.

tunnel: sofia-dev  # <-- SUBSTITUA pelo NOME do seu túnel
credentials-file: /home/mesh/.cloudflared/a1b2c3d4-e5f6-7890-1234-567890abcdef.json # <-- O cloudflared encontra isso sozinho, mas é bom saber que existe.

# Regras de entrada (ingress rules)
ingress:
  # Regra 1: Frontend Principal (SOFIA)
  - hostname: sofia.seudominio.com  # <-- SUBSTITUA pelo seu subdomínio desejado
    service: http://localhost:5173

  # Regra 2: Frontend do Agente Condensador
  - hostname: condenser.seudominio.com # <-- SUBSTITUA pelo seu subdomínio desejado
    service: http://localhost:5174

  # Regra 3: Frontend do Aquômetro
  - hostname: aquometro.seudominio.com # <-- SUBSTITUA pelo seu subdomínio desejado
    service: http://localhost:5175

  # Regra 4: Frontend do Condenser PRO (o novo )
  - hostname: condenser-pro.seudominio.com # <-- SUBSTITUA pelo seu subdomínio desejado
    service: http://localhost:5176

  # Regra Final (Catch-all ): Deve ser a última regra.
  # Qualquer tráfego que não corresponda às regras acima receberá um erro 404.
  - service: http_status:404
Salve e Feche o Arquivo: Pressione Ctrl + X, depois Y e Enter.
Configure o DNS no Cloudflare: Para cada hostname que você definiu (ex: sofia.seudominio.com ), você precisa criar um registro CNAME no painel de DNS do seu Cloudflare, apontando para o ID do seu túnel.
Tipo: CNAME
Nome: sofia (ou condenser, aquometro, etc.)
Conteúdo: a1b2c3d4-e5f6-7890-1234-567890abcdef.cfargotunnel.com (substitua pelo seu ID de túnel seguido de .cfargotunnel.com).

Python
"tunnel": [str(CLOUDFLARED_PATH), "tunnel", "--no-autoupdate", "run", "--token", str(CLOUDFLARE_TOKEN)],
Substitua essa linha pela nova versão, que é muito mais simples:
Python
# VERSÃO ATUALIZADA - Usa o arquivo de configuração padrão
"tunnel": [str(CLOUDFLARED_PATH), "tunnel", "--no-autoupdate", "run"],
Por que funciona? Quando você executa cloudflared tunnel run sem especificar um nome ou token, ele automaticamente procura e usa o arquivo config.yml no diretório ~/.cloudflared/.
Finalizando

Bash
sudo systemctl restart sofia.service
Resultado:
Após o reinício, o sentinel.py iniciará um único processo cloudflared. Esse processo lerá seu config.yml e começará a rotear o tráfego dos seus subdomínios públicos para as portas locais corretas (5173, 5174, 5175, 5176).

Você está absolutamente certo em procurar o config.yml. Essa é a maneira correta e profissional de gerenciar múltiplos serviços com o Cloudflare Tunnels. O fato de você não o encontrar significa que o túnel provavelmente está sendo executado em modo "ad-hoc" (linha de comando direta), o que explica por que ele só expõe um serviço por vez.
Vamos resolver isso de uma vez por todas, criando a configuração correta. Este processo lhe dará controle total para expor quantos serviços você quiser.
Plano de Ação: Configurando o Cloudflare Tunnels para Múltiplos Serviços
Vamos seguir um processo de 3 passos:
Localizar o Túnel: Encontrar o ID do túnel que já está em uso.
Criar o Arquivo de Configuração: Criar o config.yml e definir as regras para expor todos os seus frontends.
Atualizar o sentinel.py: Modificar o comando para que ele use o novo arquivo de configuração.
Passo 1: Localizar o ID do seu Túnel Existente
Primeiro, precisamos saber o nome ou o ID do túnel que o sentinel.py está tentando usar.
Abra um terminal e execute o seguinte comando:
Bash
cloudflared tunnel list
A saída será algo como:
Plain Text
ID                                   NAME        CREATED              CONNECTIONS
=====================================================================================
a1b2c3d4-e5f6-7890-1234-567890abcdef  sofia-dev   2025-09-20T18:00:00Z 2xDFW, 2xLAX
Anote o NOME do túnel (neste exemplo, sofia-dev). É este nome que usaremos.
Passo 2: Criar o Arquivo config.yml
Agora, vamos criar o arquivo de configuração que define todas as suas "regras de tráfego".
Navegue até o Diretório de Configuração: O local padrão para este arquivo é ~/.cloudflared/.
Bash
cd ~/.cloudflared/
(Se o diretório não existir, crie-o com mkdir ~/.cloudflared e depois entre nele).
Crie e Edite o Arquivo:
Bash
nano config.yml
Cole o Seguinte Conteúdo no Arquivo:
(Substitua sofia-dev pelo nome do seu túnel e seudominio.com pelo seu domínio real).
YAML
# Arquivo de configuração para o túnel 'sofia-dev'
# Define as regras de entrada para múltiplos serviços locais.

tunnel: sofia-dev  # <-- SUBSTITUA pelo NOME do seu túnel
credentials-file: /home/mesh/.cloudflared/a1b2c3d4-e5f6-7890-1234-567890abcdef.json # <-- O cloudflared encontra isso sozinho, mas é bom saber que existe.

# Regras de entrada (ingress rules)
ingress:
  # Regra 1: Frontend Principal (SOFIA)
  - hostname: sofia.seudominio.com  # <-- SUBSTITUA pelo seu subdomínio desejado
    service: http://localhost:5173

  # Regra 2: Frontend do Agente Condensador
  - hostname: condenser.seudominio.com # <-- SUBSTITUA pelo seu subdomínio desejado
    service: http://localhost:5174

  # Regra 3: Frontend do Aquômetro
  - hostname: aquometro.seudominio.com # <-- SUBSTITUA pelo seu subdomínio desejado
    service: http://localhost:5175

  # Regra 4: Frontend do Condenser PRO (o novo )
  - hostname: condenser-pro.seudominio.com # <-- SUBSTITUA pelo seu subdomínio desejado
    service: http://localhost:5176

  # Regra Final (Catch-all ): Deve ser a última regra.
  # Qualquer tráfego que não corresponda às regras acima receberá um erro 404.
  - service: http_status:404
Salve e Feche o Arquivo: Pressione Ctrl + X, depois Y e Enter.
Configure o DNS no Cloudflare: Para cada hostname que você definiu (ex: sofia.seudominio.com ), você precisa criar um registro CNAME no painel de DNS do seu Cloudflare, apontando para o ID do seu túnel.
Tipo: CNAME
Nome: sofia (ou condenser, aquometro, etc.)
Conteúdo: a1b2c3d4-e5f6-7890-1234-567890abcdef.cfargotunnel.com (substitua pelo seu ID de túnel seguido de .cfargotunnel.com).
Passo 3: Atualizar o sentinel.py
Agora, o passo final. Vamos dizer ao sentinel.py para parar de usar o comando antigo e passar a usar o nosso novo arquivo de configuração.
Abra o sentinel.py para edição.
Encontre a linha que define o comando do Cloudflare. Ela deve ser algo como:
Python
"tunnel": [str(CLOUDFLARED_PATH), "tunnel", "--no-autoupdate", "run", "--token", str(CLOUDFLARE_TOKEN)],
Substitua essa linha pela nova versão, que é muito mais simples:
Python
# VERSÃO ATUALIZADA - Usa o arquivo de configuração padrão
"tunnel": [str(CLOUDFLARED_PATH), "tunnel", "--no-autoupdate", "run"],
Por que funciona? Quando você executa cloudflared tunnel run sem especificar um nome ou token, ele automaticamente procura e usa o arquivo config.yml no diretório ~/.cloudflared/.
Finalizando
Salve o sentinel.py modificado.
Reinicie o serviço SOFIA para que ele use o novo comando:
Bash
sudo systemctl restart sofia.service
Resultado:
Após o reinício, o sentinel.py iniciará um único processo cloudflared. Esse processo lerá seu config.yml e começará a rotear o tráfego dos seus subdomínios públicos para as portas locais corretas (5173, 5174, 5175, 5176).
Você terá todos os seus frontends expostos de forma segura e organizada.
